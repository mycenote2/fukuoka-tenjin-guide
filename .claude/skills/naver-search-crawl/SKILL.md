# 네이버 검색 크롤링 → 브랜드 후보 발굴

새 지역 쇼핑 가이드를 만들기 전에 **"거기 가면 뭘 사나"를 네이버 후기에서 통계로 뽑아내는** 절차다.
발굴한 브랜드를 실제 가이드에 넣는 건 `add-fukuoka-brand` 스킬이 이어받는다.

이 스킬의 핵심 사고방식: **스크립트는 브랜드를 확정하지 않는다.** 후보를 수백 개에서 수십 개로
좁히고 예문을 붙여줄 뿐이고, "이건 브랜드, 이건 잡음"의 최종 판별은 클로드가 예문을 읽고 한다.
그래서 후보 목록에 노이즈가 좀 섞이는 건 정상이고 의도된 것이다. 노이즈가 무서워 필터를 조이면
정작 발굴하려던 신규 브랜드가 같이 죽는다.

## 파이프라인 3단계

```
1) naver_crawl.py     검색 목록 -> 각 글 본문 페치 -> JSONL   (기본: 키 없이 크롤)
2) extract_brands.py  형태소 분석 -> 브랜드 후보 랭킹 + 예문 -> CSV
3) 클로드가 판별       CSV의 예문을 읽고 진짜 브랜드만 추려 사용자에게 보고
```

## 사전 준비 (최초 1회)

**형태소 분석기** — `pip3 install kiwipiepy` (설치 완료됨, 2026-07-17 기준 0.23.2). 이거 하나면 된다.

**(선택) 네이버 검색 API 키** — 기본 크롤 모드는 키가 **필요 없다.** `--mode api`로 오픈 API를 쓰고
싶을 때(news/web 소스가 필요하거나 대량 수집이 안정적이길 원할 때)만 발급한다.
https://developers.naver.com/apps/#/register 에서 앱 등록 (사용 API: `검색` / 환경: `WEB 설정`에
아무 URL). 무료, 하루 25,000건. 발급값을 repo 루트 `.env.local`에 넣는다(`.gitignore`가 막아 커밋 안 됨):

```
NAVER_CLIENT_ID=발급받은_아이디
NAVER_CLIENT_SECRET=발급받은_시크릿
```

키는 **사용자가 직접** `.env.local`에 넣게 한다. 채팅에 시크릿을 붙여넣게 하지 말 것.

## 1단계: 수집

```bash
# 기본 — 키 없이 (blog/cafe만 지원)
python3 .claude/skills/naver-search-crawl/scripts/naver_crawl.py \
  --query "일본 시부야 쇼핑" \
  --query "시부야 쇼핑리스트" \
  --source blog,cafe --limit 100 --sort sim \
  --out research/naver/shibuya.jsonl
```

- **두 가지 모드** (`--mode`):
  - `scrape` (기본, 키 불필요): search.naver.com을 페이지네이션하며 링크를 긁는다. **blog/cafe만** 된다.
    실측상 8페이지 211건까지 중복 0·차단 없이 수집됐다. 제목·날짜·작성자는 목록엔 없지만 본문 페이지에서 채운다.
  - `api` (`--mode api`, 키 필요): 네이버 검색 오픈 API. **news/web 소스**까지 되고 대량에 안정적.
    어차피 본문은 따로 페치하므로 얻는 이점은 크지 않다 — 키가 있고 news를 훑고 싶을 때만 쓰면 된다.
- `--query`는 **여러 번 쓴다.** 키워드 하나로는 커버리지가 얕다. 같은 지역을 각도를 바꿔 3~5개
  (`○○ 쇼핑`, `○○ 쇼핑리스트`, `○○ 편집샵`, `○○ 득템`, `○○ 쇼핑몰 추천`) 돌리면 후보가 확 넓어진다.
  링크 기준으로 중복 제거되므로 겹쳐도 안전하다.
- `--source`: scrape는 `blog`(주력) / `cafe`. api는 여기에 `news`(신규 오픈·입점 소식에 강함) / `web`(티스토리·브런치) 추가.
- `--sort`: `sim`=정확도순(기본), `date`=최신순. **최근 뜨는 브랜드**를 찾으면 `date`.
- `--limit`: 키워드×소스당 건수. 기본 100. (scrape는 페이지당 ~25건씩, api 상한 1000)
- 출력 JSONL 필드: `source, query, title, link, description, postdate, author, body, body_status`.
- 실측 성공률(2026-07 시부야, scrape): **블로그 본문 100%**, 공개 카페만 열려 카페는 ~20%(비공개는 401, 정상).

실행이 끝나면 **본문 성공률**을 반드시 확인한다. 80% 아래면 뭔가 깨진 것이니 원인부터 본다
(스크립트가 실패 사유를 집계해서 출력한다).

## 2단계: 브랜드 후보 추출

```bash
python3 .claude/skills/naver-search-crawl/scripts/extract_brands.py \
  research/naver/shibuya.jsonl --top 60 --min-df 3 \
  --out research/naver/shibuya_brands.csv
```

- 랭킹 1순위는 **DF(문서빈도 = 몇 개의 서로 다른 글에 나왔나)**다. 한 글에서 30번 반복되는 말보다
  30개 글에 하나씩 나오는 말이 브랜드일 확률이 훨씬 높다. TF는 보조 지표.
- `kind` 열: `ko`=고유명사(돈키호테), `en`=영문(PARCO), `ja`=일본어(ロフト),
  `phrase`=붙어있는 고유명사를 이은 구(시부야 스크램블 스퀘어).
- `pair` 열: 본문에 `빔즈(BEAMS)`처럼 한글-영문 병기가 있으면 채워진다. **브랜드라는 아주 강한 신호**다.
- `--min-df`: 문서 수가 많으면(200건+) 4~5로 올려 노이즈를 줄인다.

## 3단계: 최종 판별 — 클로드가 하는 일

CSV의 `context1`/`context2`(실제 예문)를 읽고 판별한다. 예문이 결정적이다:

- `단톤` 예문이 *"베이프 살로몬 몽벨 단톤 쇼핑후기"* → **브랜드 확정**. 게다가 예문에서
  `살로몬`·`몽벨`까지 덤으로 건진다. **예문 속 동반 브랜드를 꼭 훑을 것** — 랭킹에 못 든 브랜드가 여기 숨어있다.
- `후드` 예문이 *"유니폼, 후드티, 장갑, 모자"* → 잡음. 탈락.
- `퍼플` 은 단독이면 잡음이지만 `노스페이스 퍼플` 이면 퍼플라벨 라인 → 살린다.

판별 결과는 **표로 사용자에게 보고**한다: 브랜드명 / 등장 글 수 / 무슨 가게인지 한 줄 / 근거 예문.
확실치 않은 건 "미확정"으로 따로 묶어서 보여주고 사용자 판단을 받는다. 추측으로 확정하지 말 것.

지명·일반어가 계속 상위에 올라오면 `references/stopwords.txt`에 추가한다. **새 지역을 크롤할 때마다
그 지역 지명을 여기 넣어두면 다음 실행부터 조용해진다** (예: 오사카를 크롤하면 신사이바시·도톤보리·난바 추가).

## 알아둘 함정 (실측으로 확인한 것들)

**① `blog.naver.com/{id}/{logNo}` 를 그냥 GET하면 본문이 0자다.**
2.8KB짜리 iframe 껍데기만 온다. 반드시 `m.blog.naver.com/{id}/{logNo}`로 바꿔야
`div.se-main-container`에 본문이 들어있다. (스크립트가 처리하지만, 직접 손댈 일이 있으면 기억할 것)

**② 어느 모드든 목록엔 본문이 없다.** 그래서 본문을 따로 페치한다(제목·날짜·작성자도 본문 페이지에서 채운다).
본문 페치가 실패하면 스크립트가 요약(있으면)으로 폴백하고 `body_status`에 사유를 남긴다.
scrape 모드는 검색결과 HTML의 클래스명이 자주 바뀌어서, 셀렉터 대신 **안 바뀌는 URL 패턴으로 링크만** 뽑는다.

**③ 카페 본문은 공개 카페만 열린다.** 공개 카페면 비로그인으로도 `apis.naver.com` 본문 API가 열리지만
(주소명 → 숫자 clubId 변환 필요, 스크립트가 처리), 가입전용 카페는 실패한다. 카페는 블로그보다
성공률이 낮은 게 정상이다.

**④ 정규식으로 한글 브랜드를 뽑으려 하지 말 것.** 시도했다가 상위권이 `있는/좋은/다른/있어서`로
뒤덮였다. 조사를 떼려 하면 더 망가진다 — `유니클로`의 끝 `로`, `마루이`의 끝 `이`가 조사로 보여서
`유니클`, `마루`가 되어버린다. 외래어 브랜드는 로/이/도로 끝나는 게 흔해 단순 조사 제거는 반드시
브랜드를 잘라먹는다. 그래서 Kiwi 형태소 분석기를 쓴다. 사전에 없는 브랜드(꼼데가르송·어반리서치·
니코앤드·도큐핸즈)도 고유명사로 잡아낸다.

**⑤ 구글맵 주소 블록이 최대 노이즈원이다.** 블로그에 지도를 붙여넣으면 `1 Chome−12−18`, `일본 〒`,
`Udagawachō` 같은 주소 토막이 대량 유입된다. 숫자(SN)와 비일본어 기호를 구 조합에서 빼서 막아뒀다.

**⑥ 예의.** 개인 조사용 기초자료 수집이다. `--delay`/`--workers` 기본값(0.3초, 4스레드)을 함부로
올리지 말고, 수집한 **본문 원문을 그대로 재배포하지 않는다.** 우리가 쓰는 건 "어떤 브랜드가 몇 번
언급됐나"라는 사실 통계지 남의 글 자체가 아니다.

## 다음 단계로 연결

발굴된 브랜드를 실제 가이드에 넣을 땐 **`add-fukuoka-brand` 스킬**로 넘어간다. 그 스킬이 층·주소·
좌표 교차검증과 place_id 처리를 담당한다. 이 스킬의 산출물은 어디까지나 **후보 목록**이지,
검증된 매장 정보가 아니다. 블로그 후기의 층·영업시간은 오래돼서 틀린 경우가 많으니 그대로 믿지 말 것.
