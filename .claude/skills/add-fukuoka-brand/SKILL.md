# 쇼핑 가이드에 브랜드/매장 추가하기

쇼핑 산책 가이드에 새 브랜드(또는 매장)를 정확하게 추가하기 위한 절차다. 이 가이드의 가치는 **주소·좌표·층이 실제와 정확히 일치하는 것**에 있으므로, 추측으로 채우지 말고 조사·교차검증한 값만 넣는다.

**지역이 여러 개다.** 어느 지역 요청인지 먼저 확정하고 해당 폴더만 건드린다:
`site/fukuoka-tenjin/` · `site/shibuya/` · `site/shinjuku/` (구조·스크립트 전부 동일).

## 데이터가 있는 곳

- 파일: `site/<region>/index.html` (순수 HTML/JS, 프레임워크 없음)
- 핵심 데이터 구조 (파일 안에서 아래 주석으로 검색해 위치 확인 — 줄 번호는 바뀔 수 있음):
  - `const spots = [` — 매장/건물 목록. 각 항목이 지도 핀 1개 + 카드 1개.
  - `/* 브랜드별 실제 매장 좌표 ... */ const brandCoords` — `"매장명-브랜드명": [lat, lng]`
  - `/* 매장(건물) Google Place ID ... */ const spotPlaceIds` — `"매장명": placeId`
  - `/* 브랜드별 Google Place ID ... */ const brandPlaceIds` — `"매장명-브랜드명": placeId`
- 편집 후 배포: repo 루트에서 `git add -A && git commit && git push origin main` → Vercel이 자동 배포.

## 스팟(spot)의 두 가지 유형

가이드는 매장을 **spots 배열의 한 항목 = 지도 핀 1개**로 관리한다. 두 종류가 있다:

1. **건물형 스팟** (id 1~6: 미나텐진·원후쿠오카빌딩·파르코·비오로·솔라리아·이와타야). `name`이 건물/몰 이름이고, `floors` 배열 안에 층마다 여러 `brands`가 들어간다. 대형몰·백화점 입점 브랜드가 여기 담긴다.
2. **거리형 개별 스팟** (id 7~18: 노스페이스·휴먼메이드·나나미카 등). `name`이 곧 매장명이고, 자기 좌표·주소·핀을 가진 독립 매장. `floors`는 보통 한 층에 한 브랜드.

## 절차

### 1. 입력 확보

사용자에게서 최소한 **① 매장/브랜드 이름 ② 지역(후쿠오카 텐진 등)** 을 받는다. 있으면 좋은 것: 구글맵 공유 링크, 플러스코드, 몇 층인지. 플러스코드를 주면 `scripts/decode_pluscode.py`로 좌표를 얻는다(아래 "플러스코드" 참고).

### 2. 조사 & 교차검증 (가장 중요)

WebSearch/WebFetch로 그 매장의 **정확한 위치·층·주소·좌표·영업시간**을 조사한다.

**층·주소·좌표는 반드시 2개 이상 소스로 교차검증한다.** 자동 요약(WebFetch)이 뽑은 숫자 하나만 믿지 말 것 — 요약이 층을 틀리게 뽑는 일이 실제로 있었다(라시크 스탠다드프로덕트를 B1F로 오판 → 실제 9F). 브랜드/몰 공식 매장 페이지, 구글지도, 백화점 층별안내, 지역 언론/프레스릴리스 중 최소 두 곳이 일치하는지 확인한다. 값이 엇갈리면 사용자에게 물어본다.

조사로 확정할 값: 정확한 주소, 위도/경도, 층, 영업시간, 그리고 **이 매장이 대형몰/백화점 안에 있는지 길거리 단독인지**.

### 3. 분류 — 어디에 넣을지 결정

조사 결과로 세 갈래 중 하나를 고른다:

| 상황 | 처리 |
|---|---|
| **A. 이미 가이드에 있는 건물 안** (파르코·비오로·솔라리아·이와타야·미나텐진·원후쿠오카) | 그 스팟의 해당 `floors[층].brands` 배열에 **브랜드명만 추가**. 새 핀 없음. |
| **B. 가이드에 없는 몰/백화점 안** | 그 **몰을 새 건물형 스팟으로 신규 생성**(파르코처럼 `floors` 구조). 브랜드를 해당 층에 넣음. |
| **C. 길거리 단독 매장** | 8~18번처럼 **자기 좌표·주소·핀을 가진 새 개별 스팟**으로 추가. |

새 스팟을 만드는 경우(B·C) `id`는 현재 최댓값+1로 부여한다. 지리적으로 텐진 건물 무리에 속하면 목록 순서상 6번 근처가 자연스럽지만, 중간 삽입은 뒤 id를 다 밀어야 하므로 **기본은 맨 뒤 append**(동선을 짜면 방문순서로 다시 번호가 매겨지므로 기본 번호 순서는 덜 중요하다). 순서가 중요하면 사용자에게 확인.

### 4. 항목 작성 (필드 레퍼런스)

기존 항목의 톤(반말·친근체)을 그대로 따른다. 필드:

- `id`: 정수, 스팟 고유번호 = 지도 기본 핀 번호 겸 카드 순서.
- `name` / `en`: 한글명 / 영문명.
- `lat`, `lng`: 위도/경도 (건물형이면 건물 좌표).
- `addr`: 한글 주소 ("후쿠오카시 주오구 ...").
- `access`: 접근 설명 ("니시테츠 텐진역과 바로 연결" 등).
- `hours`: 영업시간.
- `walk`: 직전 스팟 기준 도보 표현 ("솔라리아 플라자 → 도보 1분").
- `floors`: `[{ floor:"9F", brands:["브랜드A","브랜드B"] }, ...]`. `badge`는 선택("규슈 유일 매장" 등 눈에 띄는 문구).
- `note`: 한 줄 팁/설명.
- `sub`: (선택) 별관/인근 부속 브랜드 `{ title, desc, brand }`.

**케이스 A**(기존 건물에 브랜드만 추가)라면 위 필드는 안 만들고, 해당 spot의 `floors`에서 맞는 층을 찾아 `brands` 배열에 문자열 하나만 추가한다. 없는 층이면 `floors`에 새 `{floor, brands}` 객체 추가.

### 4-1. 층(floor) 표기 규칙 — **밴드 금지, 층별 1항목**

이 가이드의 핵심 가치는 "몇 층으로 가면 되는지"다. 따라서 **다층 건물은 층마다 항목을
하나씩** 만들고, 그 층에 실제로 있는 브랜드만 담는다.

```js
// ✅ 좋음 — 층마다 항목. 어느 층에 뭐가 있는지 바로 안다
floors:[
  { floor:"1F", brands:["샤넬","디올","데코르테"] },
  { floor:"3F", brands:["꼼데가르송","메종 마르지엘라"] },
  { floor:"4F", brands:["프라다","보테가 베네타"], badge:"인터내셔널 럭셔리" },
]
// ❌ 나쁨 — 여러 브랜드를 한 밴드에 뭉갬. 몇 층인지 알 수 없다
floors:[ { floor:"B1~7F", brands:["데코르테","슈에무라","꼼데가르송"] } ]
```

**밴드(`1~3F`, `B1~5F`)가 허용되는 유일한 경우는 한 브랜드가 실제로 그 층들을 통으로
쓸 때**다 (유니클로 1~3F, 무인양품 B1~3F, 핸즈 2~8F). 서로 다른 브랜드를 밴드로 묶지 않는다.

**층 소스 우선순위**

| 순위 | 소스 | 비고 |
|---|---|---|
| ① | 몰/백화점 **공식 플로어가이드** | 가장 정확. 층별 카테고리가 한 페이지에 다 있는 경우가 많다 |
| ② | **구글맵 주소 필드의 층 표기** | 개별 테넌트에 강함 (`…フラッグスビル 9～10F`). 아래 gmaps 스킬 참고 |
| ③ | 못 구하면 **층을 단정하지 말 것** | `"관내"`, `"타임스퀘어관"` 처럼 건물/구역 라벨로 두고, 나중에 확보되면 분해 |

③이 핵심이다. **모르는 층을 추측해서 숫자로 적지 않는다.** 틀린 층은 없는 층보다 나쁘다.
[[brand-data-verify-floor]] 대로 층은 2개+ 소스 교차검증이 원칙이고, 공식+구글맵이
엇갈리면 사용자에게 확인한다.

**공식 플로어가이드 수집 팁 (실제로 막혔던 것들)**

- 백화점은 **층별 개요 페이지 1개**에 전 층이 다 들어있는 경우가 많다. 층마다 8번
  페치하지 말고 개요부터 찾는다 (예: 이세탄 `…/store/shinjuku/shops.html`).
- **WebFetch가 타임아웃되는 사이트가 있다**(mistore.jp, takashimaya.co.jp 확인됨).
  이때는 브라우저로 `navigate` 후 `get_page_text` 하면 잘 읽힌다.
- 플로어가이드가 **이미지 맵**이면 텍스트 추출이 안 된다(다카시마야). → ② 구글맵으로 폴백.
- 리다이렉트되는 공식 URL이 있다(`shinjuku.hands.net` → `store.hands.net`). WebFetch가
  알려주면 새 URL로 다시 부른다.
- 검색으로 URL 패턴부터 찾는 게 빠르다(`"<몰이름> フロアガイド 各階"`).

### 5. 지도 링크는 항상 "가게 카드"로 (좌표 핀 금지)

브랜드명을 누르면 구글맵에서 **가게 카드**가 떠야 한다. `brandSearchUrl()`이 3단계로 처리한다:
1. `brandPlaceIds["매장명-브랜드명"]`(브랜드 전용 place_id) → 그 브랜드 가게 카드 (가장 정확)
2. 없으면 `spotPlaceIds["매장명"]`(그 매장/건물 place_id) → 단일매장은 그 가게, 백화점 입점은 백화점 카드. **확정적이라 동명 타지역 가게로 새지 않는다.**
3. 둘 다 없을 때만 좌표앵커 이름검색(`/maps/search/이름/@위도,경도`). 이건 이름이 강하게 겹치는 **다른 지역 가게로 샐 수 있어**(예: "세컨드스트리트"가 서울 매장으로 뜬 적 있음) 최후수단이다.

**그래서 규칙 (앞으로 브랜드 추가 시 반드시):**
- **새 개별 스팟(케이스 C)**: 그 매장 place_id를 `spotPlaceIds["매장명"]`에 **반드시** 넣는다 → 브랜드 링크가 2단계로 정확히 뜬다.
- **새 건물/몰 스팟(케이스 B)**: 그 건물 place_id를 `spotPlaceIds`에 넣는다. 개별 입점 브랜드까지 정확히 뜨게 하려면 `brandPlaceIds["매장명-브랜드명"]`도 넣는다. place_id를 못 구하면 3단계(좌표앵커)로 폴백되므로, **구글맵에서 각 브랜드 링크가 실제 그 가게로 뜨는지 직접 확인**한다(동명 타지역 주의).
- **기존 건물에 브랜드만 추가(케이스 A)**: `brandPlaceIds`에 그 브랜드 place_id를 넣으면 브랜드 카드로, 안 넣으면 건물 카드로 뜬다.
- 정확 좌표를 아는 브랜드는 `brandCoords["매장명-브랜드명": [lat,lng]]`도 넣어두면 좌표앵커가 정확해진다.

**place_id 얻는 법 (구글맵 공유 링크 처리 포함)**:

구글맵 공유 링크(`maps.app.goo.gl/...`, `goo.gl/maps/...`)는 **단축 URL이라 그 자체엔 좌표·place_id가 없다.** 링크 문자열만 보면 인식 못 하는 게 당연하니, 반드시 브라우저로 열어 실제 가게로 펼친 뒤 추출한다.

**중요:** 공유 링크가 펼쳐지는 `/maps/place/` 페이지에는 **ChIJ 형식 place_id가 없다.** hex ftid만 있으므로 변환해야 한다. (`/maps/search/` 결과 목록 페이지엔 ChIJ가 있지만, 질의가 한 곳에만 매치되면 place 페이지로 리다이렉트돼 결국 없다.) 그래서 아래 순서가 확실하다:

1. 브라우저(`mcp__Claude_Browser__preview_start` 또는 `navigate`)로 공유 링크를 연다 → 단일 가게로 펼쳐진다.
2. `get_page_text`로 **이름·주소·층·영업시간·플러스코드**를 읽는다. (층이 여기 "N층 · ..." 으로 나온다 — 단 교차검증은 필수)
3. `javascript_tool`로 **hex ftid와 좌표**를 뽑는다:
   ```js
   const h = document.documentElement.innerHTML;
   ({ ftid: h.match(/!1s(0x[0-9a-f]+:0x[0-9a-f]+)/)?.[1],   // !5s 아님! 반드시 !1s
      coord: h.match(/!3d(-?\d+\.\d+)!4d(-?\d+\.\d+)/)?.slice(1,3) })
   ```
4. ftid를 place_id로 변환:
   ```bash
   python3 scripts/ftid_to_placeid.py "0x...:0x..."
   python3 scripts/ftid_to_placeid.py --selftest   # 알고리즘 의심되면 검증
   ```
5. 검증: `https://www.google.com/maps/search/?api=1&query=<이름>&query_place_id=<ChIJ...>`를 열어 맞는 가게가 뜨는지 눈으로 확인한다.
6. 확인된 place_id를 `brandPlaceIds["매장명-브랜드명"]`(개별 입점 브랜드) 또는 `spotPlaceIds["매장명"]`(건물·단독매장)에 넣는다.

즉 **공유 링크 → 브라우저로 열기 → 이름·주소·층·좌표 읽기 → `!1s` hex ftid 추출 → ChIJ 변환 → 검증 → 저장** 순서다. (2026-07-16 라시크 3개 브랜드, 2026-07-17 몽벨을 이 방법으로 정확 카드 처리함.)

### 6. 사용자 확인

편집 전에 확정 안 된 선택지를 짧게 확인한다:
- 같은 층 복합매장이면 형제 브랜드도 같이 넣을지 (예: 스탠다드프로덕트 + DAISO + THREEPPY).
- 새 건물이면 건물 표기명 (예: "라시크 (후쿠오카 미쓰코시)").
- 목록 순서(맨 뒤 vs 중간 삽입).
- 지도 링크에 상호명 뜨게 할지(Place ID 조사 필요).

사용자가 "알아서 해줘" 하면 추천 기본값(형제 브랜드 포함 / 자연스러운 건물명 / 맨 뒤 append / 좌표 폴백)으로 진행한다.

### 7. 편집·배포·검증

1. `site/<region>/index.html` 편집.
   - **1~3곳 추가**면 Edit로 직접 삽입한다.
   - **10곳 이상 추가하거나 floors를 대대적으로 재작성**할 땐 Edit를 반복하지 말고
     **JSON 데이터 + 스플라이스 스크립트** 방식을 쓴다. `spots` / `spotPlaceIds` /
     `brandPlaceIds` 3개 블록을 정규식으로 통째 교체하는 스크립트를 scratchpad에 만들고,
     "JSON 수정 → 재실행"으로 반복한다. 300줄짜리 Edit보다 훨씬 안전하고, 스팟 수·브랜드
     수·place_id 수를 매번 출력해 검증까지 된다. (신주쿠 44곳 작업에서 검증된 방식)
     - 블록 경계 정규식: `const spots = \[.*?\n\];` / `const spotPlaceIds = \{.*?\n\};` /
       `const brandPlaceIds = \{.*?\n\};` (DOTALL, non-greedy)
     - JSON을 손으로 편집하면 **쉼표 누락**이 잦다. 스플라이스 전에 `json.loads`로 파싱해 검증.
2. 로컬 확인: `cd site && python3 -m http.server 8899` 후 `/<region>/` 에서 확인한다.
   DOM으로 한 번에 점검하면 빠르다 — 스팟 수 = 핀 수 = 카드 수, 체크리스트 개수,
   **좌표앵커 폴백이 0인지**(`a[href*="/maps/search/"]:not([href*="query_place_id"])`), 콘솔 에러 0.
   좌표가 전부 그 지역 바운딩박스 안인지도 같이 본다.
3. 커밋·푸시: repo 루트에서 `git add -A && git commit -m "..." && git push origin main`.
4. 배포 반영 확인: `curl -s "https://shopping.forijuni.com/<region>/?v=$(date +%s)" | grep "브랜드명"` 으로 폴링.
   Vercel 자동배포라 보통 10~30초 걸린다(첫 폴링은 옛 버전이 잡히는 게 정상).

## 플러스코드(Plus Code) 좌표 변환

사용자가 구글맵 플러스코드(예: `HCQ2+85 후쿠오카시`)를 주면 `scripts/decode_pluscode.py`로 위도/경도를 얻는다. 네트워크 불필요:

```bash
python3 scripts/decode_pluscode.py "HCQ2+85" --ref-lat 33.5902 --ref-lng 130.4017
```

`--ref-*`는 짧은 코드를 복원할 기준 위치(후쿠오카 텐진 근방 기본값 내장). 전체 코드(예: `8Q5GHCQ2+85`)면 기준 없이도 디코딩된다. 출력된 좌표를 구글맵(`https://www.google.com/maps?q=lat,lng`)으로 한 번 눈으로 확인한 뒤 사용한다.
