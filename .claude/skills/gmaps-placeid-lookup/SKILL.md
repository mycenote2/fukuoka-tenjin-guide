# 구글맵 Place ID 일괄 조사·정리

지역과 브랜드명 목록을 받아, 각 브랜드를 구글맵에서 검색해 **매장 후보를 나열 → 사용자가 고름 → 그
매장의 Place ID(ChIJ...)·확인/공유 링크와 구글맵 표시 정보(주소·층·건물·카테고리·영업상태·평점)를
마크다운 표로 정리**하는 스킬이다. 이 스킬은 **조사와 정리만** 한다. 결과를 실제
가이드(`site/fukuoka-tenjin/index.html`)에 넣거나 커밋·배포하는 건 `add-fukuoka-brand` 스킬의
일이다 — 사용자가 "가이드에 넣어줘"라고 하면 그쪽으로 넘긴다.

## 언제 쓰나 / 안 쓰나

- **쓴다:** "이 브랜드들 place id 찾아줘", "지역+목록 줄게 구글맵에서 찾아 표로 정리", "공유링크 뽑아줘".
- **안 쓴다(→ add-fukuoka-brand):** "가이드에 추가해줘", "이 가게 넣어줘" 처럼 가이드 편집이 목적일 때.
  (add-fukuoka-brand에도 place_id 조사 절차가 들어있지만, 그건 편집·배포까지 하는 무거운 스킬이다.)

## 핵심 원리 (왜 이 방식인가)

- 구글맵 검색 결과 페이지(`/maps/search/`)의 **각 결과 링크 href 에는 `!1s0x...:0x...` 형식 hex
  ftid 가 들어있다.** 이건 결과마다 확실히 매핑돼서(이름 ↔ ftid) 신뢰할 수 있다.
- 그 hex ftid 를 `scripts/ftid_to_placeid.py` 로 **ChIJ Place ID 로 정확히 변환**한다(네트워크 불필요,
  결정적 알고리즘). 검색 목록 페이지에 보이는 ChIJ 와 일치함을 검증했다.
- 최종 링크 `https://www.google.com/maps/search/?api=1&query=<이름>&query_place_id=<ChIJ>` 는
  **항상 그 가게 카드로 여는 안정적 permalink** 라, place_id 검증용이자 공유링크 역할을 겸한다.
  (반면 `maps.app.goo.gl` 단축 공유링크는 문자열만으론 파싱 불가라 우리가 만들지 않는다.)

## 준비물 (이 스킬 폴더 안)

- `scripts/extract_candidates.js` — 브라우저에서 검색 결과 후보를 뽑는 JS. 결과마다
  **이름·ftid + 주소·층·카테고리·영업상태·평점·전화·플러스코드**를 구조화해 반환(목록/단일 리다이렉트 모두).
- `scripts/ftid_to_placeid.py` — hex ftid → ChIJ 변환 (`--selftest` 로 알고리즘 검증 가능).
- `scripts/build_table.py` — 고른 후보 JSON → place_id + 상세정보 마크다운 표 생성(빈 컬럼 자동 생략).

## 절차

### 1. 입력 확보

사용자에게서 **① 지역(예: 후쿠오카 텐진) ② 브랜드명 목록**을 받는다. 브랜드마다 검색 질의는
`"<브랜드명> <지역>"` 로 만든다 (예: `무인양품 후쿠오카`, `몽벨 텐진`). 특정 지점을 원하면 몰/랜드마크를
붙이면 좁혀진다 (예: `스타벅스 텐진 츠타야`).

### 2. 브라우저로 검색 (브랜드마다 반복)

`mcp__Claude_Browser__preview_start`(첫 회) 또는 `navigate` 로 아래 URL을 연다. 질의는
URL 인코딩한다.

```
https://www.google.com/maps/search/<URL인코딩된 "브랜드 지역">
```

### 3. 후보 추출

`scripts/extract_candidates.js` 내용을 `javascript_tool` 로 실행한다. 반환:

```
{ mode:'list'|'place'|'empty', count, results:[{
    name, ftid, addr, floor, category, hours, phone, plusCode, rating, reviews, text }] }
```

- `mode:'list'` — 여러 매장이 매치돼 결과 피드가 뜬 경우. 피드를 끝까지 스크롤해 후보를 다 모으고, 각
  결과 카드에서 위 필드를 파싱한다.
- `mode:'place'` — 한 곳만 강하게 매치돼 단일 페이지로 리다이렉트된 경우. 그 한 곳이 답. 상세페이지의
  라벨 버튼(`data-item-id`)에서 주소·전화·플러스코드 등을 더 깔끔히 뽑는다.
- `mode:'empty'` — 못 찾음. 질의를 바꿔 다시(로마자/일본어 상호명, 지역 표기 조정).

`addr` 는 구글맵 표기 주소(건물명 포함), `floor` 는 거기서 뽑은 층이다. **둘 다 구글맵 표기 그대로라
부정확할 수 있다** — 자동 파싱값을 맹신하지 말고, `text`(카드/상세 요약)를 **클로드가 읽어** 지점명·주소·층을
사람이 알아볼 수 있게 확인·정리한다. 층은 [[brand-data-verify-floor]] 대로 가이드 반영 시 교차검증 필요.

> **대형 체인 주의:** 구글맵은 결과를 **현재 지도 뷰포트(줌 레벨)** 기준으로 준다. 브랜드명이 특정
> 매장과 강하게 매치되면 그 매장으로 확 줌인(17z)해 후보가 1개만 나오기도 한다. 이 임베디드
> 브라우저에서는 줌아웃·"이 지역 검색" 같은 지도 조작이 신뢰성 있게 먹지 않는다. 그러니 스타벅스·다이소
> 처럼 지점이 많은 체인에서 특정 지점을 원하면 **질의를 더 구체화**한다(몰·건물·랜드마크 추가). 무인양품처럼
> 지역 내 매장이 2~3곳인 경우는 그대로 후보가 다 나온다.

### 3-1. 층(floor) 뽑기 — 정규식 함정 4가지

구글맵 **주소 필드에 층이 같이 들어있다**(`新宿3-37-1 フラッグスビル 9～10F`). 개별 테넌트의
층을 알아내는 가장 빠른 방법이라, 공식 플로어가이드가 없는 건물은 이걸로 층을 채운다
(→ `add-fukuoka-brand`의 "층 소스 우선순위 ②").

다만 **주소 문자열에서 층을 정규식으로 뽑을 때 실제로 틀렸던 케이스가 4가지** 있다.
전부 겪은 것이니 아래 패턴을 그대로 쓴다:

```js
// F 뒤에 알파벳이 오면 층이 아님(Flags/Floor 오탐 방지), M접두(메자닌) 허용
const FLG=/(地下\s*[0-9０-９]+\s*階|[MＭBＢ][0-9０-９]+\s*[FＦ](?![a-zA-Z])|[0-9０-９][0-9０-９\-~～,，\s]*\s*(?:階|[FＦ](?![a-zA-Z])))/g;
const floors = (addr||'').match(FLG);   // 전체 매치를 보존해서 사람이 확인
```

| 함정 | 실제 사례 | 결과 | 대응 |
|---|---|---|---|
| **단어의 F를 층으로 오탐** | `3-37-1 Flags, 4-5F` | `1F`(Flags의 F) ← 틀림. 실제 `4-5F` | `[FＦ](?![a-zA-Z])` |
| **메자닌 M 접두 누락** | `NEWoMan新宿 M2F` | `2F` ← 층 이름이 다름. 실제 `M2F` | 패턴에 `[MＭ]` 포함 |
| **번지수를 층으로 오탐** | `千駄ヶ谷5-24-2 1階` | `2` 를 주움. 실제 `1階` | 매치 여러 개면 **뒤쪽**이 층. 원문 확인 |
| **범위가 쪼개짐** | `2F～6F` | `2F`, `6F` 로 분리 | 매치를 `join`해 원문 보존, 범위로 기록 |

**결론: 자동 파싱값을 그대로 데이터에 넣지 말 것.** 반드시 `addr`/`text` **원문을 클로드가 읽고**
층을 확정한다. 층은 [[brand-data-verify-floor]] 대로 가이드 반영 시 교차검증 대상이다.

### 3-2. 결과 오염 필터 (스폰서 광고 · 좌표 범위)

- **스폰서 광고 주의:** 체인·중고매입 계열을 검색하면 구글맵이 **전혀 다른 지역의 광고**를
  첫 결과로 끼워넣는다(실제로 "코메효 신주쿠" 검색에 네리마 소재 매입점 광고가 떴다).
  카드 텍스트에 `스폰서`/`Sponsored` 가 있으면 **버린다.**
- **좌표 바운딩박스 검증:** 뽑은 좌표가 대상 지역 범위 밖이면 다른 지역 매장이다. 버리고
  질의를 좁힌다. (신주쿠 예: `lat 35.680~35.706, lng 139.688~139.716`)
- 필터 후 결과가 0개면 **없는 것으로 처리하고 추측해 넣지 않는다.** 질의를 바꿔 재시도하거나
  사용자에게 확인한다.

### 3-3. 좌표(lat/lng)도 같이 뽑기

`extract_candidates.js` 는 place_id 조사용이라 **좌표를 반환하지 않는다.** 가이드에 스팟을
새로 넣으려면 지도 핀용 좌표가 필수이므로, 각 결과 href에서 같이 뽑는다:

```js
const C=h=>{let m=h.match(/!3d(-?\d+\.\d+)!4d(-?\d+\.\d+)/);   // 우선
            if(!m) m=h.match(/@(-?\d+\.\d+),(-?\d+\.\d+)/);     // 폴백
            return m?[+m[1],+m[2]]:[null,null];};
```

### 4. 후보 제시 → 사용자 선택

브랜드마다 뽑힌 후보를 **번호를 매겨 이름·주소·층**으로 사용자에게 보여주고 어느 지점인지 고르게 한다
(사용자가 "후보 다 나열하고 내가 고른다"를 택했다). 후보가 1개면 그대로 확정. 애매하면(동명 타지역 등)
표시해 확인받는다.

### 5. 표 생성

고른 후보들을 JSON 배열로 만들어 `build_table.py` 에 넘긴다. 각 항목 최소 `{name, ftid}`, 있으면
`addr, floor, category, hours, rating, reviews, phone, plusCode`(extract 결과를 그대로 넘기면 됨).

```bash
echo '[{"name":"MUJI (JR博多시티)","ftid":"0x354191c7ec9f313d:0x589c0dfc213c97d8","addr":"Hakataekichuogai, 1−1 ＪＲ博多シティ ６Ｆ","floor":"6F","category":"잡화점","hours":"영업 종료 · 오전 10:00에 영업 시작","rating":"4.0","reviews":"661"}]' \
  | python3 scripts/build_table.py --title "후쿠오카 텐진"
```

출력은 `매장/브랜드 | 주소 | 층 | 카테고리 | 영업 | 평점 | place_id | 확인·공유 링크` 마크다운 표다.
**값이 하나도 없는 상세 컬럼은 자동 생략**된다. `--phone` 을 붙이면 전화, `--pluscode` 를 붙이면
플러스코드 컬럼이 추가된다. 여러 브랜드를 한 번에 조사했으면 각 항목을 배열에 담아 한 번 호출해 한 표로 낸다.

### 6. 검증

표의 **확인·공유 링크 중 최소 몇 개**를 브라우저로 열어 실제 그 가게 카드(이름·주소·층 일치)로 뜨는지
확인한다. 동명 타지역 가게로 새지 않았는지 특히 본다. place_id 는 확정적이라 좌표앵커 검색과 달리 잘
안 새지만, 후보 선택이 틀렸을 수 있으니 눈으로 확인한다.

## 출력 예시

```
### 후쿠오카 (무인양품) — Place ID

| 매장/브랜드 | 주소 | 층 | 카테고리 | 영업 | 평점 | place_id | 확인·공유 링크 |
|---|---|---|---|---|---|---|---|
| MUJI (JR博多시티) | Hakataekichuogai, 1−1 ＪＲ博多シティ ６Ｆ | ６Ｆ | 잡화점 | 영업 종료 · 오전 10:00 시작 | 4.0 (661) | `ChIJPTGf7MeRQTUR2Jc8IfwNnFg` | [지도 열기](https://www.google.com/maps/search/?api=1&query=MUJI&query_place_id=ChIJPTGf7MeRQTUR2Jc8IfwNnFg) |

> 주소·층은 구글맵 표기 기준이라 부정확할 수 있다. 가이드 반영 시 2개+ 소스로 교차검증할 것.
```

## add-fukuoka-brand 로 넘기기

사용자가 이어서 "이걸 가이드에 넣어줘" 하면, 정리한 place_id 를 `add-fukuoka-brand` 절차로 넘긴다:
`brandPlaceIds["매장명-브랜드명"]`(개별 입점 브랜드) 또는 `spotPlaceIds["매장명"]`(건물·단독매장)에 넣는다.
그때 층·주소·좌표는 그 스킬 규칙대로 **2개 이상 소스로 교차검증**한다.

## 완료 체크리스트

- [ ] 브랜드마다 `"브랜드 지역"` 으로 검색했는가
- [ ] 후보를 이름·주소·층으로 사용자에게 제시하고 선택받았는가 (1개면 확정)
- [ ] ftid → ChIJ 변환값이 있고, 표에 place_id + 확인·공유 링크 + 주소·층 등 상세정보가 들어갔는가
- [ ] 확인 링크 몇 개를 열어 실제 그 가게로 뜨는지 검증했는가 (동명 타지역 주의)
- [ ] **스폰서 광고를 걸러냈는가** (`스폰서`/`Sponsored` 카드는 타지역 매장일 수 있음)
- [ ] **좌표가 대상 지역 바운딩박스 안에 있는가** (밖이면 다른 지역 매장이니 버림)
- [ ] **층은 정규식 결과를 그대로 쓰지 않고 원문(addr/text)을 읽어 확정했는가** (Flags의 F·M2F·번지수·범위 오탐 주의)
- [ ] (가이드에 새 스팟으로 넣을 거면) **좌표 lat/lng도 href에서 같이 뽑았는가**
- [ ] 주소·층은 구글맵 표기라 부정확할 수 있음을 표에 명시했는가 (footnote 자동 출력)
- [ ] (가이드 삽입 요청 시) add-fukuoka-brand 로 넘겼는가 — 이 스킬은 편집·배포 안 함
